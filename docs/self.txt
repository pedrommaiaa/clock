# Clock — Project Summary

## What Clock Is

Clock is a **modular AI-assisted engineering system** built on **Unix philosophy**.

Its purpose is to create a **transparent, composable, and safe alternative to monolithic coding agents** like:

* Claude Code
* OpenClaw
* Devin-style autonomous agents
* Codex CLI tools

Instead of a single large application, Clock is a **collection of tiny tools** that work together through structured pipelines.

Each tool:

* does **one job**
* communicates via **stdin → stdout**
* uses **structured JSON / JSONL**
* can be combined like **Unix pipes**

Clock becomes a **programmable operating environment for AI-assisted software development**.

---

# Core Philosophy

Clock follows three main principles.

## 1. Unix Philosophy

The system is built from **small programs** that do one thing well.

Examples:

* `srch` → search code
* `slce` → extract file slices
* `pack` → build prompt context
* `guard` → validate patches
* `aply` → apply diffs
* `vrfy` → run verification

These tools compose together:

```
srch → slce → pack → llm → act → guard → aply → vrfy
```

Instead of one giant application, Clock behaves like **a programmable shell ecosystem**.

---

## 2. Transparency

Every step is visible and auditable.

Clock logs everything:

* tool calls
* LLM requests
* patch proposals
* verification results
* system traces

All stored as:

```
.clock/trce.jsonl
```

This makes the system:

* debuggable
* replayable
* explainable
* safe

---

## 3. Safety First

Clock enforces strict safety rules:

* **all code changes must be patches**
* **patches must pass validation**
* **verification runs after every change**
* **rollback always available**
* **high-risk changes require approval**

The system never silently mutates code.

---

# What Problem Clock Solves

Modern AI coding tools often have problems:

### Lack of transparency

You cannot see what the agent is actually doing.

### Hard to control

Agents run complex loops internally.

### Difficult to debug

When something fails, it is unclear why.

### Vendor lock-in

Tools are tied to a single LLM provider.

### Unsafe autonomy

Agents can accidentally damage repositories.

---

Clock solves these problems by building an **open, inspectable, programmable system**.

It acts more like a **Unix toolchain for AI development** than a black-box agent.

# Core Capabilities

Clock enables five major capabilities.

---

# 1. AI-Assisted Development

Clock can help with tasks like:

* analyzing codebases
* suggesting improvements
* generating patches
* fixing failing tests
* summarizing projects
* refactoring modules
* generating documentation

The AI is treated as **a tool in the pipeline**, not the system itself.

---

# 2. Autonomous Engineering

Clock can run **continuously in autonomous mode**.

Examples:

* monitor CI failures
* propose fixes
* open pull requests
* refactor code periodically
* scan dependencies
* audit security issues

This is implemented using:

* `dock` (daemon)
* `watch` (event detection)
* `tick` (scheduler)
* `q` (job queue)
* `work` (workers)

---

# 3. Distributed AI Swarms

Clock can coordinate **multiple AI agents** working together.

Examples of roles:

* planner
* editor
* reviewer
* tester
* security auditor
* performance optimizer

The system orchestrates them through:

```
swrm
role
hub
farm
```

This enables more complex reasoning than a single LLM loop.

---

# 4. Long-Term Knowledge

Clock builds a persistent understanding of projects.

It stores:

* repository structure
* API contracts
* module relationships
* successful fixes
* recurring problems

This is managed through:

```
knox (knowledge graph)
mem  (durable memory)
note (long-term summaries)
pbk  (playbooks)
```

This allows Clock to **learn from past work**.

---

# 5. Self-Improvement

Clock can analyze its own behavior and improve its capabilities.

This includes:

* detecting inefficiencies
* generating new tools
* testing improvements
* benchmarking performance
* promoting successful tools

This process uses:

```
diag
self
spec
forge
test
bench
gate
prom
roll
```

This creates a **controlled evolution mechanism**.

The kernel is the trusted core.

* safety policies
* artifact hashing
